## Problem Statement

Belochka's frontend UI is entirely in English. Users who are more comfortable in Chinese, French, or Russian must navigate an interface they may not fully understand. As a monitoring tool intended for diverse teams, the lack of language support limits accessibility and usability for non-English-speaking operators.

## Solution

Add internationalization (i18n) to the React frontend using `react-i18next`. Support four languages: English (default/fallback), Chinese (Simplified), French, and Russian. The UI language is auto-detected from the browser, persisted in `localStorage`, and manually switchable via a language selector visible on all pages.

## User Stories

1. As a Chinese-speaking operator, I want the dashboard to display in Chinese, so that I can monitor servers without language barriers.
2. As a French-speaking operator, I want to see all UI labels, buttons, and messages in French, so that I can work more efficiently.
3. As a Russian-speaking operator, I want the interface in Russian, so that I can understand all controls and status messages at a glance.
4. As a first-time visitor, I want the app to automatically detect my browser language preference and display in my language, so that I don't need to configure anything.
5. As a user whose browser language is not supported (e.g. Japanese), I want the app to fall back to English, so that I still get a usable interface.
6. As a user, I want to manually switch the UI language via a dropdown menu, so that I can override the auto-detected language.
7. As a user, I want my language choice to persist across browser sessions, so that I don't have to re-select my language every time.
8. As a user on the Dashboard page, I want to see the language switcher, so that I can change the language at any time.
9. As a user on the Server Detail page, I want to see the language switcher, so that I can change the language without navigating back to the dashboard.
10. As a user, I want the language switcher to display each language in its native name (e.g. "Francais" not "French"), so that I can find my language regardless of the current UI language.
11. As a user, I want all dialog titles, form labels, and button texts to be translated, so that I can add, edit, and delete servers in my language.
12. As a user, I want toast notifications (success/error messages) to appear in my chosen language, so that I understand what happened after an action.
13. As a user, I want the "Connection lost, reconnecting..." banner to appear in my language, so that I understand the connection status.
14. As a user, I want the empty state ("No servers configured") to display in my language, so that I understand what to do next.
15. As a user, I want server detail labels (Hostname, Kernel, Uptime, OS, Cores) to display in my language, so that I can read system information clearly.
16. As a user, I want section headings (CPU, Memory, Disk, Network, Processes) to display in my language.
17. As a user, I want the "Server not found" error page to display in my language.
18. As a user, I expect technical units (KiB, MB/s, %, d/h/m) to remain unchanged regardless of language, so that they stay consistent with industry conventions.
19. As a user, I expect backend error messages (SSH connection failures, API errors) to remain in English, since they contain technical details that are language-independent.

## Implementation Decisions

- **i18n library**: `react-i18next` with `i18next` core and `i18next-browser-languagedetector` for automatic language detection.
- **Supported languages**: `en` (English, fallback), `zh` (Chinese Simplified), `fr` (French), `ru` (Russian).
- **Translation file structure**: One JSON file per language, stored in a dedicated i18n directory within the frontend source. No namespace splitting -- all keys in a single file per language. Approx. 50 keys per file.
- **Key naming convention**: Dot-separated, grouped by page/component. Examples: `dashboard.title`, `dashboard.noServers`, `addServer.name`, `addServer.testConnection`, `serverDetail.cpu`, `serverDetail.hostname`. Truly shared strings (Save, Cancel, Delete, Close) use a `common.*` prefix.
- **Language detection order**: `localStorage` (user's prior choice) -> `navigator` (browser language) -> fallback `en`. This is the default behavior of `i18next-browser-languagedetector`.
- **Language switcher**: A new global Layout component wraps all routes in App.tsx, providing a top bar with the language dropdown in the right corner. The switcher is a dropdown showing native language names: English, Chinese, Francais, Russian. Uses Radix UI Select (already a project dependency) for the dropdown. Lucide `Globe` icon as visual indicator.
- **Format functions**: `formatBytes`, `formatNetworkSpeed`, `formatPercent`, `formatUptime` in `format.ts` remain untouched. Technical units (B, KiB, MB/s, %, d/h/m) are not translated.
- **Error messages**: Frontend-hardcoded strings (toast fallbacks like "Connection test failed", "Failed to save server") are translated via `t()`. Backend `err.message` values are passed through untranslated.
- **Interpolation**: Used for dynamic values in translated strings, e.g. `t('addServer.savedSuccess', { name: server.name })` renders "Server \"Production\" added successfully" or the equivalent in other languages.
- **No backend changes**: The Go backend is not modified. i18n is purely a frontend concern.

## Testing Decisions

Good tests for this feature verify externally observable behavior: that the correct translated text appears in the DOM under different language configurations, and that user interactions (switching language) produce the expected UI changes. Tests should NOT assert on internal i18n state, key lookups, or implementation details of the i18n library.

Prior art: existing component tests in `web/src/__tests__/` use `vitest` + `@testing-library/react` + `userEvent`, rendering components inside `MonitorContext` and `MemoryRouter` providers. The i18n tests follow the same pattern, adding an `I18nextProvider` wrapper.

**Modules to test:**

1. **Translation key completeness** -- A unit test that loads all four JSON translation files and asserts that `zh`, `fr`, and `ru` have exactly the same set of keys as `en`. This catches missing or extraneous keys without testing translation quality.

2. **i18n initialization** -- A unit test that verifies the i18n instance initializes with all four languages loaded and falls back to `en` for unsupported languages.

3. **Component rendering under i18n** -- Existing component tests (Dashboard, AddServerDialog, ConnectionBanner, ServerDetail, etc.) are updated to render inside the i18n provider. They continue to assert on visible text, which now comes from translations. At minimum, one test per major component verifies that switching language changes the rendered text.

4. **Language switcher** -- A new component test for the Layout/language-switcher that verifies: (a) the dropdown renders with four language options, (b) clicking a language updates the displayed UI text, (c) the selection persists (i18next stores it in localStorage).

## Out of Scope

- **Backend i18n**: Go backend error messages, API responses, and log output remain in English.
- **RTL layout support**: No right-to-left languages are being added.
- **Translation of technical units**: Byte units, speed units, percentages, and uptime abbreviations are not translated.
- **Dynamic locale loading / code splitting**: All translations are bundled statically. Lazy loading translation files is unnecessary at this scale.
- **Pluralization rules**: The current UI strings do not require plural forms (e.g. "3 cores" is formatted programmatically, not via i18n plurals). If future strings need plurals, i18next supports them natively.
- **Date/number locale formatting**: Numbers and dates are not reformatted per locale (e.g. decimal comma vs. period).
- **Translation management tooling**: No integration with external translation management platforms (Crowdin, Lokalise, etc.).

## Further Notes

- Russian has three plural forms (one, few, many). The current UI does not have strings that require pluralization, but if added in the future, i18next handles Russian plurals natively.
- The `Globe` icon from Lucide (already a project dependency) is a natural fit for the language switcher trigger.
- Existing tests that match UI text via `getByText` or `getByRole({ name: ... })` will need their matchers updated to use the English translation values, or the test i18n instance should be configured to use `en` explicitly.
