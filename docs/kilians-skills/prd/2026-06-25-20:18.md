## Problem Statement

Belochka users can monitor server metrics in the browser, but when they need to investigate an issue or perform ad-hoc operations on a server, they must switch to a separate terminal application and manually SSH in. This context switch is disruptive — they lose sight of the dashboard and must manage SSH credentials independently. Users need a way to open an interactive shell session directly from the monitoring interface.

## Solution

Add a web-based interactive terminal (SSH console) accessible from each server's detail page. A "Console" button opens a new browser tab with a full terminal emulator connected to the server via the existing SSH connection pool. Users get a complete shell experience (supports interactive programs like vim, htop, etc.) without leaving the Belochka interface.

## User Stories

1. As a server operator, I want to click a "Console" button on a server's detail page, so that I can open an interactive terminal to that server without switching applications.
2. As a server operator, I want the console to open in a new browser tab, so that I can view the monitoring dashboard and terminal side by side.
3. As a server operator, I want to see the server's name, host address, and authentication method (password/key) displayed in the console page, so that I know which server I'm connected to.
4. As a server operator, I want to use interactive programs (vim, htop, less, nano) in the web terminal, so that I have the same capabilities as a native SSH client.
5. As a server operator, I want the terminal to automatically adjust to my browser window size, so that full-screen programs render correctly when I resize the window.
6. As a server operator, I want to see a clear "disconnected" message when the SSH connection drops, so that I know my session has ended.
7. As a server operator, I want a "Reconnect" button when the connection drops, so that I can start a new shell session without reloading the page.
8. As a server operator, I want to open multiple console tabs for the same server, so that I can run different tasks in parallel.
9. As a server operator, I want the terminal session to be cleaned up when I close the browser tab, so that orphaned shell sessions don't accumulate on my servers.
10. As a server operator, I want the console to reuse the existing SSH connection used for monitoring, so that I don't need to wait for a separate authentication handshake.
11. As a server operator, I want terminal input/output to feel responsive with minimal latency, so that typing and command output feel natural.
12. As a server operator, I want the console page to display correctly across different screen sizes, so that I can use it on various monitors.
13. As a server operator, I want the Console button to be visually distinct from the Delete button, so that I don't accidentally click the wrong action.
14. As a server operator, I want the Console button label and status messages to be translated into all supported languages (English, Chinese, French, Russian), so that the feature is consistent with the rest of the UI.

## Implementation Decisions

### Backend — new `internal/terminal` module

- A new `internal/terminal` package handles terminal session lifecycle: opening a PTY-enabled SSH session, bridging stdin/stdout between the WebSocket and the SSH session, and processing resize requests.
- The module defines a `SessionOpener` interface for obtaining an SSH session from a server ID. The SSH pool implements this interface by calling `client.NewSession()` on the existing managed connection.
- Each WebSocket connection maps to exactly one SSH session with a requested PTY (term type "xterm-256color").

### Backend — WebSocket endpoint

- New endpoint: `GET /api/ws/terminal/{serverID}` — upgrades to WebSocket, creates an SSH session with PTY, and bridges data bidirectionally.
- The endpoint is registered in the chi router alongside existing routes, injected via a new `RouterOption` (e.g., `WithTerminalSessionOpener`).
- Message protocol:
  - **Terminal data** (both directions): binary WebSocket frames containing raw bytes. No JSON wrapping, no base64.
  - **Control messages** (text WebSocket frames):
    - Client to server: `{"type":"resize","cols":<int>,"rows":<int>}`
    - Server to client: `{"type":"status","status":"connected"|"disconnected","message":"..."}`
- On WebSocket close (browser tab closed), the handler closes the SSH session and PTY.
- On SSH session EOF or error, the handler sends a `disconnected` status message and closes the WebSocket.

### Backend — SSH Pool extension

- Add a method to the SSH Pool that exposes session creation: given a server ID, return a new `*gossh.Session` from the existing managed `*gossh.Client`.
- This method is the concrete implementation behind the `SessionOpener` interface.

### Backend — graceful shutdown

- Terminal sessions are closed during the existing shutdown sequence, before the SSH pool is closed. Add a shutdown step that closes all active terminal sessions.

### Frontend — new console page

- New route: `/server/:id/console` — renders a full-page terminal.
- The page header displays: server name, host address (host:port), and auth type (password/key). This data is fetched via the existing `GET /api/servers/{id}` REST endpoint.
- Terminal emulator: xterm.js (`@xterm/xterm`) with the fit addon (`@xterm/addon-fit`).
- WebSocket connection: plain browser `WebSocket` API (not the existing `WebSocketProvider`), connecting to `/api/ws/terminal/{serverID}`.
- On resize: `ResizeObserver` on the terminal container triggers `fitAddon.fit()`, which sends a resize JSON message over the WebSocket.
- On disconnect: overlay the terminal with a "Disconnected" message and a "Reconnect" button. Reconnect creates a new WebSocket + new terminal instance.

### Frontend — console entry button

- In `ServerDetail.tsx`, add a "Console" button in the title bar, positioned to the left of the existing "Delete" button.
- The button uses the `Terminal` icon from lucide-react and primary/default styling.
- `onClick` opens `/server/:id/console` in a new tab via `window.open`.
- Button label uses i18n key for translation support.

### i18n

- Add translation keys for: console button label, page title, connection status messages, reconnect button, auth type labels. Add to all four language files (en, zh, fr, ru).

## Testing Decisions

### What makes a good test

Tests should verify external behavior through the module's public interface, not internal implementation details. They should be deterministic (no real SSH connections, no timing dependencies) and use the same patterns established in the codebase.

### Backend tests

- **`internal/terminal` module tests**: Inject a mock `SessionOpener` that returns a mock SSH session (in-memory pipe for stdin/stdout). Verify:
  - PTY is requested with correct term type and initial window size.
  - Data written to the mock stdout appears as binary WebSocket frames sent to the client.
  - Binary WebSocket frames from the client are written to the mock stdin.
  - Resize JSON messages trigger `WindowChange` calls with correct dimensions.
  - WebSocket close triggers session cleanup.
  - SSH session EOF sends a `disconnected` status message.

- **WebSocket handler tests**: Follow the `hub_test.go` pattern — `httptest.NewServer` with gorilla WebSocket client. Inject a mock `SessionOpener`. Verify:
  - Successful upgrade and `connected` status message.
  - Invalid server ID returns appropriate error.
  - Binary frame round-trip through mock session.

### Frontend tests

- No unit tests for the console page. The terminal rendering and WebSocket interaction are best verified through manual browser testing. The existing codebase has no frontend test infrastructure, and xterm.js rendering cannot be meaningfully tested in jsdom.

### Prior art

- `internal/hub/hub_test.go`: WebSocket endpoint testing with `httptest.NewServer` + gorilla client — same pattern for terminal WebSocket tests.
- `internal/monitor/collector_test.go`: Interface-based mocking for SSH execution — same pattern for `SessionOpener` mock.

## Out of Scope

- Shell session persistence or resumption (tmux/screen integration) — a disconnected session is lost.
- File upload/download through the terminal interface.
- Shared/collaborative terminal sessions (multiple users viewing the same shell).
- Terminal session recording or audit logging.
- Customizable terminal appearance (font size, color scheme).
- SSH key passphrase support (already listed as a separate future consideration).
- Rate limiting or authentication/authorization for the terminal endpoint (the app has no auth layer currently).

## Further Notes

- The `xterm.js` library and its fit addon are new npm dependencies that will be added to the frontend.
- The terminal feature reuses the SSH connection pool's reconnection and keepalive logic — if the pool reconnects after a network interruption, the user can manually reconnect the terminal to get a new session on the restored connection.
- The Console button should only be visually enabled; no special handling is needed for servers in "reconnecting" state — the WebSocket handler will return an error if the SSH connection is unavailable, and the frontend will display the disconnected state.
