## Problem Statement

System administrators managing 5-20 remote Linux servers need a centralized way to monitor resource usage (CPU, memory, disk, network) and running processes in real time. Currently this requires SSH-ing into each server individually and running diagnostic commands, which is time-consuming, error-prone, and provides no unified overview of fleet health.

## Solution

Belochka is a single-binary web application that maintains persistent SSH connections to configured servers, collects system metrics every 2 seconds, and streams them to a browser-based dashboard via WebSocket. Administrators add servers through a web UI, and immediately see live resource gauges, usage bars, and process tables — all from a single browser tab.

## User Stories

1. As an administrator, I want to add a server by entering its SSH credentials in a web form, so that I can start monitoring it without editing config files.
2. As an administrator, I want to test an SSH connection before saving a server, so that I know the credentials are correct.
3. As an administrator, I want to see the host key fingerprint during connection testing, so that I can verify I'm connecting to the right host (Trust on First Use with explicit confirmation).
4. As an administrator, I want to authenticate with either password or SSH key file, so that I can use whichever auth method my servers require.
5. As an administrator, I want to edit a server's configuration (host, port, credentials, display name), so that I can update it when things change.
6. As an administrator, I want the system to require re-testing the connection when I change connection-related fields, so that I don't save broken configurations.
7. As an administrator, I want to change only a server's display name without re-testing, so that cosmetic changes are frictionless.
8. As an administrator, I want to delete a server, so that I stop monitoring decommissioned machines.
9. As an administrator, I want to see a dashboard grid of all my servers with CPU, memory, disk, and network stats on each card, so that I get a fleet-wide overview at a glance.
10. As an administrator, I want server cards to show color-coded usage bars (green 0-60%, yellow 60-80%, red 80-100%), so that I can instantly spot servers in trouble.
11. As an administrator, I want the dashboard to show the disk partition with the highest usage on each card (with its mount point name), so that I see the most critical disk situation per server.
12. As an administrator, I want the dashboard to show aggregated network throughput from physical interfaces only (filtering out lo, docker*, veth*, br-*, virbr*), so that I see meaningful traffic numbers.
13. As an administrator, I want to click a server card to navigate to a detailed monitoring page, so that I can investigate a specific server.
14. As an administrator, I want the detail page to show CPU usage as a ring gauge plus per-core horizontal progress bars, so that I can see overall and per-core load.
15. As an administrator, I want the detail page to show memory usage as a ring gauge including swap information, so that I can assess memory pressure.
16. As an administrator, I want the detail page to show all disk partitions (excluding virtual filesystems) with usage bars, so that I can check every mount point.
17. As an administrator, I want the detail page to show per-interface network throughput (all interfaces), so that I can identify which interface carries traffic.
18. As an administrator, I want the detail page to show the top 20 processes sorted by CPU, with clickable column headers to re-sort by memory/PID/user, so that I can find resource-hungry processes.
19. As an administrator, I want the detail page to show system info (hostname, kernel, uptime, OS, core count), so that I have context about the machine.
20. As an administrator, I want metrics to update every 2 seconds without manual refresh, so that I see near-real-time data.
21. As an administrator, I want disconnected servers to show a status message in their card (at the same size and position) instead of stale data, so that I know which servers are unreachable.
22. As an administrator, I want to see a banner when the WebSocket connection to Belochka is lost, with the last data visually dimmed, so that I know the dashboard is stale.
23. As an administrator, I want the WebSocket to auto-reconnect with exponential backoff, so that temporary network issues resolve without manual intervention.
24. As an administrator, I want SSH connections to auto-reconnect with exponential backoff (capped at 30 seconds), so that transient network issues self-heal.
25. As an administrator, I want the system to detect dead SSH connections (3 consecutive collection failures) and trigger reconnection, so that zombie connections don't silently produce gaps.
26. As an administrator, I want non-retryable SSH errors (auth failure) to stop reconnection attempts and show a clear error, so that I know to fix the configuration.
27. As an administrator, I want a guided empty state when no servers are configured, so that first-time setup is obvious.
28. As an administrator, I want to run Belochka with zero configuration (no config file needed), so that I can get started immediately.
29. As an administrator, I want a dark-themed interface, so that long monitoring sessions are comfortable and metrics are visually prominent.
30. As an administrator, I want data formatted in human-readable units (GiB/MiB with 1024 base, 1 decimal place, network in KB/s or MB/s), so that numbers are immediately meaningful.
31. As an administrator, I want a single binary that serves both API and frontend, so that deployment is just copying one file.
32. As an administrator, I want server passwords encrypted at rest in the database, so that credential exposure risk is reduced.
33. As an administrator, I want a startup warning if the encryption key is auto-generated and co-located with the database, so that I'm aware of the security implication.
34. As an administrator, I want the API to never return password plaintext, so that credentials aren't leaked through the UI layer.

## Implementation Decisions

### Project Structure

Standard Go project layout with `cmd/server/main.go` as entry point and all business logic under `internal/`. Frontend lives in `web/`. Build orchestrated by Makefile with targets: `make dev-backend`, `make dev-frontend`, `make build`.

```
belochka/
├── cmd/server/main.go
├── internal/
│   ├── config/
│   ├── api/
│   ├── model/
│   ├── ssh/
│   ├── monitor/
│   ├── hub/
│   └── store/
├── web/
├── go.mod
├── Makefile
└── CLAUDE.md
```

### Go Module and Dependencies

- Module path: `belochka`
- SQLite: `modernc.org/sqlite` (pure Go, no CGO) with WAL mode enabled and single-writer serialization
- HTTP router: `chi`
- WebSocket: `gorilla/websocket`
- SSH: `golang.org/x/crypto/ssh`
- Logging: standard library `log/slog`, text format to stdout

### SSH Connection Pool

- One persistent connection per server
- Auto-reconnect with exponential backoff: 1s → 2s → 4s → ... → 30s cap
- Keepalive every 30s, mark disconnected after 3 consecutive failures
- Host key verification: Trust on First Use with UI fingerprint confirmation. Server model stores `HostKeyFingerprint` field. "Test Connection" returns the fingerprint, user must explicitly accept before saving
- No passphrase support for key-based auth; keys with passphrases are rejected at test time with a clear error
- Non-retryable errors (auth failure, host key mismatch) stop reconnection; user must fix config

### Metrics Collection

- One goroutine per server, 2-second collection interval
- All commands combined into single SSH exec per cycle (no sleep, no multi-sample within a cycle)
- CPU and network rates computed via cross-cycle deltas (current values minus previous cycle's raw values stored in memory)
- Process list via `top -bn1 -o %CPU | head -27` (procps-ng only; unsupported systems degrade to empty process list)
- Disk collection via `df -B1 -x tmpfs -x devtmpfs -x overlay -x squashfs` (virtual filesystems filtered at command level)
- Per-collection timeout: 5 seconds. Single timeout skips current cycle. 3 consecutive failures trigger SSH reconnection
- Collection failure does not disconnect SSH on its own; the consecutive-failure counter triggers reconnection

### WebSocket Hub

- Application-level singleton connection (established in top-level React provider, shared across pages)
- Full broadcast: all server metrics pushed to all clients every 2 seconds (~15-20KB for 20 servers)
- On new connection: immediately push a `snapshot` message containing server list + latest metrics
- Connection status changes pushed as `status` type messages
- Maximum 10 concurrent WebSocket connections; new connections beyond limit receive close frame (code 1013)
- Message types: `snapshot` (initial full state), `metrics` (periodic update), `status` (connection state changes)

### REST API

- CRUD for servers at `/api/servers`
- `POST /api/servers/{id}/test` for connection testing (returns host key fingerprint on first connection)
- Unified error format: `{"error": {"code": "machine_readable", "message": "Human readable"}}`
- HTTP status codes: 400 (bad input), 404 (not found), 422 (test connection failure), 500 (internal)
- Passwords never returned in API responses; empty string on edit means "keep current password"

### Server Lifecycle

- Add: form validation (non-empty fields only) → Test Connection (mandatory) → display fingerprint → user confirms trust → save to SQLite → start SSH connection → start collector goroutine
- Edit (connection fields changed): reset validation state → require re-test → re-confirm fingerprint if host changed → save → reconnect with new config
- Edit (name only): save directly, no re-test
- Delete: stop collector goroutine → close SSH connection → remove in-memory metrics → broadcast removal to WebSocket clients → delete from SQLite (database delete last as implicit "commit")

### Frontend

- React + TypeScript, Vite build tool
- shadcn/ui components: button, card, dialog, input, select, label, table, badge, progress, sonner, alert
- Tailwind CSS, dark theme only (`class="dark"` on root element)
- State: single React Context + useReducer; server cards wrapped in `React.memo`
- Routing: React Router — `/` (dashboard), `/server/:id` (detail, UUID in URL)
- Vite dev server on port 53137, proxying `/api` to Go backend on port 53136 (`ws: true` for WebSocket)
- Production: frontend built assets embedded into Go binary via `go:embed`, served on port 53136

### Dashboard UI

- Grid of server cards, ordered by `CreatedAt` (add time), position fixed regardless of connection state
- Each card: status indicator, server name, host, CPU/memory progress bars, disk (highest-usage partition with mount point label), network throughput (aggregated physical interfaces)
- Disconnected cards: same size, content replaced with status icon + message (e.g., "Reconnecting (3/∞)", "Auth failed")
- Empty state: centered prompt with "Add your first server" button opening the add-server modal

### Detail Page UI

- System info bar: hostname, kernel, uptime, OS, core count
- 2×2 grid: CPU ring gauge (conic-gradient) + per-core horizontal progress bars, memory ring gauge + swap, disk usage bars (all physical partitions), network throughput per interface (all interfaces shown)
- Process table: top 20, directly replaced on each update, clickable column headers for client-side re-sort
- Color thresholds (step function, not gradient): green 0-60%, yellow 60-80%, red 80-100% — applies to CPU, memory, disk. Network shows values only, no color coding

### Data Formatting

- Memory/disk: 1024-based units (KiB, MiB, GiB), 1 decimal place
- Network: bytes/s (KB/s, MB/s), 1 decimal place
- Percentages: 1 decimal place

### WebSocket Disconnection UX

- Top banner: "Connection lost, reconnecting..."
- Last received data retained but visually dimmed (reduced opacity overlay)
- Banner disappears and data restores on successful reconnection

### Network Interface Filtering

- Backend collects all interfaces
- Frontend filters for dashboard display: exclude `lo`, `docker*`, `veth*`, `br-*`, `virbr*`
- Detail page shows all interfaces

### Configuration

- Config file: `belochka.yaml` (optional)
- Load priority: `--config` flag → `./belochka.yaml` in CWD → built-in defaults
- Zero-configuration startup: all settings have sensible defaults (port 53136, data_dir `./data`)
- Data directory: `./data` relative to CWD (SQLite database + auto-generated encryption key)
- Encryption key: config file or `BELOCHKA_ENCRYPTION_KEY` env var; auto-generated if absent, with startup warning

### Graceful Shutdown

- `context.Context` propagated through all components, cancelled on SIGTERM/SIGINT
- Shutdown order: stop accepting HTTP connections → close WebSocket connections (send close frame) → stop all collector goroutines → close all SSH connections → close SQLite
- Hard timeout: 10 seconds, then force exit

## Testing Decisions

Good tests in this project verify external behavior through stable interfaces, not implementation details. Tests should be deterministic and fast. Mock external dependencies (SSH connections, network) but use real SQLite (in-memory mode) for store tests.

### Modules and Seams

- **Metrics parser** (`internal/monitor/parser.go`): highest-value seam. Pure functions that take command output strings and return typed `Metrics` structs. Test with captured output from various Linux distributions. No mocks needed.
- **SSH connection pool** (`internal/ssh`): test against the `Pool` interface using a mock SSH server. Verify connection state machine transitions, reconnection backoff, consecutive-failure detection, host key verification.
- **REST API handlers** (`internal/api`): use `httptest` with mocked service layer. Test CRUD operations, error responses, the test-connection flow including fingerprint return.
- **WebSocket hub** (`internal/hub`): test broadcast delivery, connection limit enforcement (11th connection rejected), snapshot push on new connection.
- **Store layer** (`internal/store`): use in-memory SQLite. Test CRUD, password encryption/decryption round-trip, WAL mode initialization.
- **Frontend format utilities**: unit test `formatBytes`, `formatPercent`, color threshold functions with Vitest.
- **Frontend components**: React Testing Library for card rendering with various metric states, disconnected state display, empty state.
- **useWebSocket hook**: mock WebSocket to test reconnection logic, snapshot handling, message type dispatching.

### What NOT to test

- Actual SSH connections to real servers
- Exact CSS styles or pixel-level layout
- React Context internal state shape
- Specific log messages

## Out of Scope

Explicitly excluded from V1, architecturally supported for future extension:

- Web terminal / SSH console
- Cron job management
- Alert notifications (email, Slack, etc.)
- Historical data storage and trend charts
- Multi-user authentication and permissions
- HTTPS / TLS termination
- Public internet deployment
- macOS or BusyBox target server support
- SSH key passphrase support
- User-configurable card ordering (drag and drop)
- Light theme or theme switching
- Client-side WebSocket subscription filtering
- Docker-native deployment (Dockerfile)

## Further Notes

- The application name "Belochka" (белочка, Russian for "squirrel") is the project codename.
- Target servers must run standard Linux distributions with procps-ng (Ubuntu, Debian, CentOS, RHEL, Fedora, etc.). Servers running BusyBox-based `top` (Alpine, embedded) will have degraded process list functionality (empty) but all other metrics will work.
- The 53136/53137 port numbers are non-standard by design to avoid conflicts with common development tools.
- The architecture is designed for extensibility: the SSH connection pool is shared infrastructure, and new features (console, cron) register as independent modules that obtain connections from the pool.
